3. LANGUAGE MODEL (LLM) - OPTIMIZED FOR 8GB RAM
================================================================================

Ollama
  └─ Version: 0.1.0+
  └─ Platform: WSL2 (Ubuntu) on Windows 64-bit
  └─ Deployment: REST API on localhost:11434 (accessible from Windows)

Recommended Model (8GB RAM Constraint):
  mistral:7b-instruct-q4_0
    └─ Parameters: 7 billion
    └─ Quantization: 4-bit (Q4_0)
    └─ RAM Usage: ~4 GB
    └─ Performance: Excellent for RAG tasks, fast inference on CPU
    └─ Why: Best balance between quality and memory efficiency

Alternative Model:
  llama3:8b-instruct-q4_0
    └─ Parameters: 8 billion
    └─ Quantization: 4-bit (Q4_0)
    └─ RAM Usage: ~4.5 GB
    └─ Performance: Slightly better accuracy, marginally slower
    └─ Why: If extra 0.5GB RAM is acceptable

⚠️ AVOID These Models (RAM Incompatible):
  └─ llama3 (non-quantized): Requires 16GB+ RAM
  └─ mistral (non-quantized): Requires 14GB+ RAM
  └─ llama3:70b-*: Requires 40GB+ RAM
  └─ Any model without "q4_0" or "q4_K_M" suffix

Why Local LLM on 8GB RAM:
  └─ Data privacy: UNIDO policy documents never leave local machine
  └─ No API costs: Free inference after initial setup
  └─ Offline operation: No internet required after model download
  └─ 4-bit quantization: Minimal quality loss (~2-3%), 75% memory savings

6. DATA-SPECIFIC REQUIREMENTS

6.1 Input File Formats
    ──────────────────
    PDF (.pdf)
      └─ Count: 3 files
      └─ Characteristics: Text-extractable (no scanned images), multi-page
      └─ Processing: PyMuPDF for text extraction
      └─ Files: UNIDO_Finance.pdf, UNIDO_HR.pdf, UNIDO_Manufacturing.pdf

    Text (.txt)
      └─ Count: 3 files
      └─ Format: Structured key-value pairs, one record per 2 lines (data + blank line)
      └─ Processing: Custom regex parsing + pandas DataFrame conversion
      └─ Files: Finance_and_Accounting.txt, Human_Resource_Management.txt, 
                Manufacturing_and_Production.txt

6.2 Data Preprocessing Requirements
    ───────────────────────────────
    Text Cleaning
      └─ Remove: Extra whitespace, blank lines, special characters (if needed)
      └─ Preserve: Semantic structure (headings, regulations, record fields)

    Missing Value Handling
      └─ HR Dataset: "Not Available" in Promotion_Type field (intentional, keep as-is)
      └─ No NaN/null replacement needed (text format already contains placeholder strings)

    Field Extraction
      └─ Finance: 9 fields (Transaction_ID, Date, Biennium, Budget_Type, Fund_Type, 
                            Category, Amount_EUR, Status, Approved_By)
      └─ HR: 11 fields (Employee_ID, Division, Gender, Grade, Category, Appointment, 
                        Hire_Date, Yrs_in_Grade, Perf_Rating, Promo_Eligible, Promotion_Type)
      └─ Manufacturing: 12 fields (Record_ID, Region, Development_Group, Industry_Sector, 
                                   Year, Quarter, IIP_Index_2015_100, MVA_Weight_Pct, 
                                   QoQ_Growth_Pct, YoY_Growth_Pct, Seasonally_Adjusted, 
                                   Data_Status)

6.3 Chunking Strategy
    ─────────────────
    PDF Documents
      └─ Method: Recursive character splitting with overlap
      └─ Chunk size: 512-1024 tokens
      └─ Overlap: 50-100 tokens (preserve context across chunks)
      └─ Reason: Multi-page policy documents (11-22 pages) exceed single-embedding limits

    Text Datasets
      └─ Method: Record-level chunking (each record = 1 chunk)
      └─ Serialization: Convert structured fields to natural language
      └─ Example: "Finance transaction FIN-0001 on 2024-04-24 in biennium 2024-2025 
                   with Regular Budget, General Fund, Appropriation of 561910.79 EUR, 
                   Approved status by Conference"
      └─ Reason: Preserve complete record context for accurate retrieval

8. DEPLOYMENT ARCHITECTURE
================================================================================

8.1 Pipeline Components (in execution order)
    ────────────────────────────────────────

    [INGESTION PHASE]
      1. PDF Text Extraction (PyMuPDF)
           └─ Input: 3 UNIDO policy PDFs
           └─ Output: Raw text strings

      2. Text Dataset Parsing (pandas + regex)
           └─ Input: 3 text files (300 records)
           └─ Output: Structured DataFrames

      3. Text Serialization
           └─ Input: DataFrames
           └─ Output: Natural language descriptions per record

      4. Text Chunking (LangChain splitter)
           └─ Input: PDF text + serialized records
           └─ Output: ~500-1000 context-preserving chunks

      5. Embedding Generation (sentence-transformers)
           └─ Input: Text chunks
           └─ Output: Dense vectors (384-dim or 768-dim)

      6. Vector Storage (ChromaDB)
           └─ Input: Embeddings + metadata
           └─ Output: Indexed vector database (persistent)

    [QUERY PHASE]
      7. Query Embedding (sentence-transformers)
           └─ Input: User natural language query
           └─ Output: Query vector

      8. Similarity Search (ChromaDB)
           └─ Input: Query vector
           └─ Output: Top-k most relevant chunks (k=3-5)

      9. Context Assembly (LangChain)
           └─ Input: Retrieved chunks
           └─ Output: Formatted context for LLM

     10. Response Generation (Ollama LLM)
           └─ Input: Query + retrieved context
           └─ Output: Grounded natural language answer

8.2 Persistence Requirements
    ────────────────────────
    ChromaDB Storage
      └─ Location: Local directory (e.g., ./chroma_db/)
      └─ Backend: SQLite (included with ChromaDB)
      └─ Persistence: Database persists across sessions (no re-indexing needed)

    Source Documents
      └─ Location: Local directory (e.g., ./data/)
      └─ Required at runtime: No (only during initial ingestion)

10. CRITICAL IMPLEMENTATION NOTES (8GB RAM OPTIMIZED)
================================================================================

10.1 Text Dataset Parsing Strategy
     ────────────────────────────
     Pattern: "Department: X. Field1: Value1. Field2: Value2. ..."
     Approach:
       1. Read file line-by-line
       2. Skip blank lines
       3. For each data line:
          - Split by ". " delimiter
          - Extract key-value pairs using regex: r"(\w+(?:\s+\w+)*): (.+)"
          - Store in dictionary
       4. Convert list of dictionaries to pandas DataFrame

10.2 Memory-Efficient Embedding Strategy (8GB RAM)
     ─────────────────────────────────────────────
     Model: all-MiniLM-L6-v2 (REQUIRED - do not use larger models)
       └─ Dimensions: 384 (vs 768 for all-mpnet-base-v2)
       └─ RAM: ~100MB (vs ~400MB for all-mpnet-base-v2)
       └─ Speed: 2x faster encoding
       └─ Quality: 95% of all-mpnet performance, sufficient for enterprise docs

     Batch Processing:
       └─ Encode chunks in batches of 32 (not all at once)
       └─ Example: model.encode(chunks[0:32]), then chunks[32:64], etc.
       └─ Reason: Prevents RAM spike during embedding generation

10.3 ChromaDB Configuration (Disk-Based, Not In-Memory)
     ──────────────────────────────────────────────────
     CRITICAL: Use persistent mode to keep vectors on disk, not RAM

     Correct Setup:
       import chromadb
       client = chromadb.PersistentClient(path="./chroma_db")
       collection = client.get_or_create_collection("unido_kb")

     ⚠️ AVOID:
       client = chromadb.Client()  # In-memory mode (loads all vectors to RAM)

     Metadata Strategy:
       └─ Collection: Single "unido_kb" collection
       └─ Metadata: {"department": "Finance", "source": "UNIDO_Finance.pdf"}
       └─ Filtering: collection.query(where={"department": "HR"})

10.4 Ollama Inference Optimization (CPU-Only, 8GB RAM)
     ─────────────────────────────────────────────────
     Environment Variables (set in WSL2 Ubuntu before running Ollama):
       export OLLAMA_NUM_PARALLEL=1        # Only 1 request at a time
       export OLLAMA_MAX_LOADED_MODELS=1   # Only 1 model in RAM
       export OLLAMA_NUM_GPU=0             # Force CPU (no GPU acceleration)

     Ollama Server Start:
       ollama serve  # Run this in Ubuntu terminal, keep it open

     From Windows Python:
       import requests
       response = requests.post(
           "http://localhost:11434/api/generate",
           json={"model": "mistral:7b-instruct-q4_0", "prompt": query}
       )

10.5 System Resource Management (8GB RAM Survival Guide)
     ───────────────────────────────────────────────────
     Before Running RAG System:
       1. Close Chrome/Edge (browsers consume 1-2GB RAM)
       2. Close VS Code, PyCharm, or other IDEs if not needed
       3. Close Spotify, Discord, Slack
       4. Check Task Manager: Aim for <4GB RAM usage before starting

     During Execution:
       └─ Python script: ~1.5GB
       └─ ChromaDB: ~500MB (disk-backed, not RAM)
       └─ Ollama LLM: ~4GB
       └─ Windows: ~2GB
       └─ Total: ~8GB (tight but functional)

     If RAM Runs Out:
       └─ Windows will use page file (disk swap)
       └─ Inference becomes slower but doesn't crash
       └─ Ensure at least 10GB free disk space for swap

10.6 WSL2 Memory Limit Configuration (OPTIONAL)
